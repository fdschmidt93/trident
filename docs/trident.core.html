class trident.core.datamodule.TridentDataModule(*args: Any, **kwargs: Any)[source]¶

Bases: lightning.

The base class for all datamodules.

The TridentDataModule facilitates writing a LightningDataModule with little to no boilerplate via Hydra configuration. It splits into

  • dataset_cfg:

  • dataloader_cfg:

Parameters

dataset_cfg (omegaconf.dictconfig.DictConfig) –

A hierarchical DictConfig that instantiates or returns the dataset for self.dataset_{train, val, test}, respectively.

Typical configurations follow the below pattern:

See also

src.utils.hydra.instantiate_and_apply(), src.utils.hydra.expand() dataloader_cfg (omegaconf.dictconfig.DictConfig):

See also

src.utils.hydra.expand()

seed (int, optional):

Linked against config.seed by default for convenience and maybe used for functionality that is not yet set by lightning.seed_everything, which sets the seed for pytorch, numpy and python.random.

Notes

  • The train, val, and test keys of dataset_cfg and dataloader_cfg join remaining configurations with priority to existing config

  • dataloader_cfg automatically generates train, val, and test keys for convenience as the config is evaluated lazily (i.e. when a DataLoader is requested)

Example

_target_: src.datamodules.base.TridentDataModule
_recursive_: false

dataset_cfg:
  _target_: datasets.load.load_dataset
  # access methods of the instantiated object
  _method_:
    map: # dataset.map for e.g. tokenization
      # kwargs for dataset.map
      function:
        _target_: src.utils.hydra.partial
        _partial_: src.datamodules.preprocessing.{...}
      num_proc: 12

  path: glue
  name: mnli
  train:
    split: "train"
  val:
    # inherits `path`, `name`, etc.
    split: "validation_mismatched+validation_matched"
  test:
    # set `path`, `name`, `lang` specifically, remainder inherited
    path: xtreme
    name: xnli
    lang: de
    split: "test"

dataloader_cfg:
  _target_: torch.utils.data.dataloader.DataLoader
  batch_size: 8
  num_workers: 0
  pin_memory: true
  # linked against global cfg
  seed: ${seed}

setup(stage: Optional[str] = None) → None[source]¶

Sets up self.dataset_{train, val, test} datasets that are fed to the corresponding DataLoader instances.

dataset_cfg must point to a function with the following specification. The below example shows a naive use case:

def setup(cfg: DictConfig, datamodule: TridentDataModule, stage: Optional[str]) -> None:
    # load and process the dataset as defined in dataset_cfg.my_dataset configuration
    dataset = hydra.utils.instantiate(cfg.my_dataset)
    datamodule.dataset_train = dataset["train"]
    datamodule.dataset_val = dataset["val"]
    datamodule.dataset_test = dataset["test"]

The corresponding datamodule configuration may then look as follows:

dataset_cfg:
    _target_: src.custom.my_setup_func
    _partial_: true

    my_dataset:
        _target_: src.custom.dataset.MyDataset
        ...
Parameters

  • self – datamodule

  • stage – one of “fit”, “validate”, “test”

See also

LightningDataModule.setup, src.datamodules.utils.load_dataset(), src.utils.hydra.instantiate_and_apply()

Mixin for base model to define evaluation loop largely via hydra.

The evaluation mixin enables writing evaluation via yaml files, here is an example for sequence classification, borrowed from configs/evaluation/classification.yaml.

# apply transformation function
prepare_cfg:
  batch: null # on each step
  outputs:    # on each step
    _target_: src.utils.hydra.partial
    _partial_: src.evaluation.classification.get_preds
    .. code-block: python

        # we link evaluation.apply.outputs against get_preds
        def get_preds(outputs):
            outputs.preds = outputs.logits.argmax(dim=-1)
            return outputs

  step_outputs: null  # on flattened outputs of what's collected from steps
# Which keys/attributes are supposed to be collected from `outputs` and `batch`
step_outputs:
  outputs: "preds" # can be a str
  batch: # or a list[str]
    - labels

# either metrics or val_metrics and test_metrics
# where the latter
metrics_cfg:
  # name of the metric used eg for logging
  accuracy:
    # instructions to instantiate metric, preferrably torchmetrics.Metric
    metric:
      _target_: torchmetrics.Accuracy
    # either on_step: true or on_epoch: true
    on_step: true
    compute:
      preds: "outputs:preds"
      target: "batch:labels"
  f1:
    metric:
      _target_: torchmetrics.F1
    on_step: true
    compute:
      preds: "outputs:preds"
      target: "batch:labels"
setup(stage: str)[source]¶

Sets up the TridentModule.

Setup the model if it does not exist yet. This enables inter-operability between your datamodule and model if you pass define a setup function in module_cfg, as the datamodule will be set up _before_ the model.

In case you pass : setup to module_cfg the function should follow the below schema:

def setup(module: TridentModule, stage: str):
    # the module has to be setup
    module.model = hydra.utils.instantiate(module.hparams.model)

Since the module exposes module.trainer.datamodule, you can use your custom setup function to enable inter-operability between the module and datamodule.

Comprises training step of your model which takes a forward pass.

Notes:

If you want to extend training_step, add a on_train_batch_end method via overrides. See: Pytorch-Lightning’s on_train_batch_end

Implementation:

def training_step(
    self, batch: BatchEncoding, batch_idx: int
) -> Union[dict[str, Any], ModelOutput]:
    outputs = self(batch)
    self.log("train/loss", outputs.loss)
    return outputs
Parameters

  • batch_idx – variable used internally by pytorch-lightning

Returns

weights_from_checkpoint(ckpt_path: str, map_location: str = 'cpu', strict: bool = True, processing_fn: Optional[Callable] = None, map_weights: Optional[Union[str, dict]] = None)[source]¶

Loads weights from existing Pytorch-Lightning checkpoint.

Parameters

  • ckpt_path (str) – path to lightning checkpoint

  • map_location (Optional[Callable]) – device to map to

  • strict (bool) – force agreement (True) or accept non-matching keys (False)

  • map_weights (Union[None, str, dict[str, str]], optional) –

    • If None: assumes 1-to-1 alignment between checkpoint and TridentModule

    • If str: maps the weight of the str to self.model

    • If dict: keys of the checkpoint’s state_dict are mapped by values to self.model

      • Values should be in absolute dot notation e.g. encoder.embeddings: transformer.embeddings

  • map_location – function that takes the state dict of the loaded checkpoint, process and outputs the state dict
